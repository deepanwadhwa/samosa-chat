## Hardware-aware context

The gateway's Auto command leaves `-c` unset, enables Prism's fitter with a
4 GiB device-memory target, uses one slot, and bounds batch/microbatch to
512/256. An explicit `-c 0` was rejected during development because Prism
interprets it as a request for the model's full trained context; it is not the
same as leaving the option unset.

| Backend | Artifact exercised | Fitted context | Exact response | Decode |
|---|---|---:|---|---:|
| Bonsai | `Bonsai-27B-Q1_0.gguf`, 3,803,452,480 bytes | 66,816 | `BONSAI AUTO OK` | 16.28 tok/s |
| Ornith | qualified local `Ornith-1.0-9B-Q4_K_M.gguf`, 5,701,067,872 bytes | 94,464 | `ORNITH AUTO OK` | 18.89 tok/s |

Both capacities were read from `/props.default_generation_settings.n_ctx`
after warmup. Neither number exists as a Samosa model tier or fixed default.

The complete Samosa gateway was then started in an isolated home with Bonsai.
`/healthz` reported `context_limit_tokens:66816` and
`context_limit_mode:"hardware-auto"`. A non-streaming request through
`/v1/chat/completions` returned exactly `GATEWAY AUTO OK` and reported
`session_saved:true`; gateway shutdown then stopped the backend cleanly.

## Safety note

Two exploratory commands that explicitly requested the full 262,144-token
context demonstrated why `-c 0` is unsafe as an Auto implementation on this
host: one produced Metal out-of-memory errors and swap growth. Those processes
were stopped. The final unset-context fitter commands completed warmup and
generation while preserving the configured device margin. This failed path is
recorded so future changes do not reintroduce `-c 0`.
