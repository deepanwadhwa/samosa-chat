# E-JI1 real-model run — Phase JI find on real Ornith 9B (2026-07-23)

First real-model exercise of the Phase JI find pipeline (increments 1–3,
`80127c6` on `issue-7-jobs`). **Not** the full 500-file E-JI1 measurement — a
scaled 14-file proof to validate the model-decides path end to end on real
Ornith before scaling up. It found a real defect the offline fake-backend tests
cannot catch, so it is recorded here in full.

## Setup

- Machine: the reference 16 GB M3 (the only machine Samosa is measured on).
- Backend: **real Ornith 9B** (`Ornith-1.0-9B-Q4_K_M.gguf`) via the installed
  `llama-server`, launched by the freshly-built `build/samosa-gateway` (my JI
  code), isolated on port 8652/8653 with a temp `SAMOSA_HOME`. Real
  `samosa-fs`/`samosa-extract`/`samosa-ocr` (built this session) + the real
  `ocr-pack-v1`. The owner's live gateway on :8642 was left untouched.
- Folder: 14 files — 12 junk-named text files (recipe, song lyrics, invoice,
  medicare/medical-coding decoys, wallpaper, todo, readme, trip, budget,
  meeting, training) + **2 planted targets**:
  - `policar_research_2019.txt` — well-named, text-layer content.
  - `CamScanner_03-15-2024.png` — anonymous phone-scan name; the subject
    ("Poličar 2019") exists **only in the pixels** (the R-series OCR fixture
    `tools/testdata/ocr/tiny.png`, which OCRs to "Poličar 2019").
- Goal: *"Find all files related to the Poličar 2019 research."*

## Result: MIXED — the pipeline works, the anonymous-scan case fails

**What worked on the real model** (all previously unverified against real Ornith
— the fake backend only scripts these):

- Model-driven filename triage (Phase A) correctly rejected all 12 junk files
  *by name* with sensible reasons ("Recipe file, unrelated.", "Song lyrics,
  unrelated.", …) and kept the well-named target as `"likely"`. The RC1
  delimiter bug's replacement — 100% model judgment — behaves sanely.
- Skim (Phase B) read the text target's first lines into `skim.jsonl`.
- Classify (Phase C) and the **Phase D verify loop + structured `finish()`**
  ran end to end: the model called `fs_read_text` then `fs_metadata` on the
  candidate, then emitted a valid `finish()` with a page-quoted evidence string
  and a coherent note. `result.json` is well-formed. The finish contract, the
  no-prose-ending rule, and the tool loop all hold on the real 9B model.

**What failed — the core scenario the card exists for:**

Real Ornith triaged the anonymous scan as **`"no"`**, dropping it before the
skim:

```
{"rel_path":"CamScanner_03-15-2024.png","verdict":"no","why":"Generic CamScanner scan, unrelated."}
```

The triage system prompt explicitly says *"unknown is the correct verdict when a
name (for example an anonymous scan like CamScanner or IMG_1234) says nothing
about content — those files are read later."* The model ignored it and excluded
the file on the exact reasoning the design forbids ("unrelated" — a content
judgment made from a name that carries no content). Consequence:

- The scan was never skimmed, never OCR'd, its pixel content never reached the
  model. `skim.jsonl` has **one** row (the text file only).
- `result.json` reports **1 match** and is **silent about the miss** —
  `rejected_count: 0`, `unreadable: []`, notes: *"Only one file was
  shortlisted."* A plural "find all" goal returned a single-file answer with no
  signal that a plausible file was dropped unread.

This is the Titli-in-pixels failure (RC5 / JI.8-b) reproduced on the real model.
The offline fake-backend suite scripts triage as `"unknown"`, so it is
structurally incapable of catching this — the gate did its job.

## Measurements (coarse — from polling, not per-event instrumentation)

- Total wall: ~111 s for 14 files. Phase A triage (one 14-file batch, one
  Ornith generation) dominated the first ~40 s; Phase D verify (2 tool rounds +
  finish) the remainder.
- Peak `llama-server` RSS: 5.8 GB. **Swap delta: 0** (used stayed 381.31 MB
  before and after). Clean backend shutdown on gateway SIGTERM. Owner's :8642
  server untouched. No thermal/machine-safety concern at this scale.
- No per-token or per-phase timing was instrumented; the full E-JI1 (≈500 files,
  cold + warm cache, per-phase wall) is still to run.

## Diagnosis

The design is sound and the model-decides verify→finish path works. The failure
is **Phase A reliability**: a 9B model conflates "anonymous/uninformative name"
with "irrelevant" and emits `"no"` where the design requires `"unknown"`. Since
the whole premise is "a scanned record with a meaningless filename must be
findable, because its content is reachable," a triage that can drop such files
by name defeats the premise. The prompt alone is not a strong enough guarantee.

Secondary: the sweep-honesty contract (design law 5) is not enforced in C — the
model was free to return a single-file answer to a plural goal and say nothing
about the dropped file. Nothing mechanically flagged the gap.

## Fix options (owner decision — touches design law 1)

1. **Prompt-only:** sharpen the anonymous-scan rule + a few-shot example; the
   model still decides. Least invasive, design-law-clean, but relies on 9B
   instruction-following (just shown fragile).
2. **Reframe triage semantics:** the model may only answer `"no"` when the name
   clearly names a *different* subject; uninformative names default to
   `"unknown"`. Still model-decided (prompt change), biased toward keeping.
3. **Mechanical safety net:** the gateway upgrades a `"no"` on a *readable* file
   with an opaque/generic-scan name (CamScanner/IMG_/DSC_/Scan/bare number or
   date) to `"unknown"`. Guarantees the scan is read, but is C pattern-matching
   on names — in tension with design law 1 (which JI.0 was built to enforce).
4. **Skim-all-readable:** triage cannot drop readable files at all; every
   readable survivor is skimmed (the content-addressed cache amortizes re-runs).
   Fully design-law-clean (a mechanical "readable files are always read" policy,
   no name interpretation), but the first skim of a large folder is the
   "tens of minutes" cost the triage filter was meant to bound.

All four should also add a mechanical sweep-honesty check so a plural goal that
finishes with dropped-unread files surfaces them.

## Fix verified — confidence triage re-run (same day, real Ornith)

The owner's chosen redesign (model assigns confidence, never hard-drops; skim
reads in confidence order within budget; files past budget are deferred, not
lost) was implemented and the identical 14-file scenario re-run on real Ornith.
**Both planted targets are now found.**

- Phase A verdicts.jsonl (confidence, no drops): the well-named target →
  `"high"`; the 12 junk files → `"low"` with per-file reasons; and the scan →
  `{"rel_path":"CamScanner_03-15-2024.png","confidence":"medium","why":
  "CamScanner image — anonymous name, must read to know content"}` — the model
  now reasons exactly as the design intends.
- Phase B skim.jsonl: the scan was OCR-skimmed —
  `{"path":"CamScanner_03-15-2024.png","confidence":"medium",
  "first_lines":"Poličar 2019","source":"ocr","deferred":false}`. `skim_progress`
  shows all 14 files skimmed **in confidence order** (high → medium → low).
- Phase D/E result.json — a complete sweep:
  ```json
  "matches":[
    {"path":"policar_research_2019.txt","evidence":"Research summary. Author: Policar. Year 2019. …","page":1,"confidence":"high"},
    {"path":"CamScanner_03-15-2024.png","evidence":"Poličar 2019","page":1,"confidence":"high"}]
  ```
  The verify loop called `fs_read_text` on the text file and `doc.read` on the
  scan, then `finish()` with both. The anonymous-scan miss is gone.
- ~127 s wall for 14 files (warm llama-server); peak `llama-server` RSS 5.8 GB;
  **swap delta 0**; clean shutdown; owner's :8642 untouched.

Still open: the full ~500-file E-JI1 (cold + warm cache, per-phase measurements)
and E-JI2 warm re-run; the Stage-2 continuity flow (a follow-up re-skims the
deferred low-confidence tail) and a `deferred` field on the result card; and a
mechanical sweep-honesty check. This scaled run is the "works" proof for the
core scenario, not the full measured gate.
